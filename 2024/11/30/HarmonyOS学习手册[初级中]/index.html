项目工程结构

项目工程结构有如下目录层级

应用项目

  • AppScope
    • resources
      • base
        • element
        • media
    • app.json5
  • entry
    • src
      • main
        • ets
          • entryability
          • pages
        • resources
          • base
            • element
            • media
            • profile
          • en_US
          • rawfile
          • zh_CN
        • module.json5
    • oh-package.json5
  • oh-modules
  • oh-package.json5

AppScope目录由DevEco Studio自动生成，不可更改。

Module目录名称可以自定义，默认由DevEco Studio自动生成如（entry、library等）

接下来会根据三个类型：ArkTS 源码文件、资源文件、配置文件对这个目录结构一一说明

ArkTS源码文件

entry -> src -> main -> ets是ArkTS源码文件目录

  • entryability是模块的Ability文件目录：其存放的UIAbility组件就是一种包含UI的应用组件，可以用于和用户交互。-

    • entryAbility.ets文件是entry模块入口的UIAbility组件

      在entryAbility.ets 文件中提供了一些生命周期回调函数供于使用，例如指定当前UIAbility组件需要加载的页面

  • pages是模块的页面目录：其存放的页面文件用于表示页面展示

    • index.ets 文件是模块的入口页面文件。我们可以通过在index.ets文件实现页面的绘制

配置文件

目录下部分配置文件路径如下

【AppScope - > app.json5】、【entry -> src -> main -> module.json5】、【entry ->oh-package.json5】、【oh-package.json5】等

配置文件可以分为两类：应用级配置文件、模块级配置文件

  • app.json5 ：应用级配置文件，提供了应用的配置信息

  • module.json5： 模块级配置文件，提供了模块的配置信息

    比如模块的基础信息、模块应用组件信息、模块权限信息等

  • oh-package.json5 ：在这里路径是entry -> oh-package.json5。模块级依赖管理配置信息

    定义了名称、版本号、作者以及代码中三方库等信息

  • oh-package.json5：在这里路径是应用项目 -> oh.package.json5 。应用级依赖管理配置信息

    其作用和模块级依赖管理配置文件类似

资源文件

资源文件分为应用级资源文件以及模块级配置文件，其具体路径如下：

应用级资源文件目录 【AppScope -> resources】 、模块级资源文件目录【entry -> src -> main -> rescources】

resources目录构成

  • element： 元素资源目录，包含数字、颜色、字符串等
  • media：媒体资源目录，包含图片、音频、视频等
  • profile：自定义配置文件目录，包含页面配置、卡片配置等配置文件
    • main_pages.json ：默认生成，定义模块页面的路由集合
  • en_US：模块内会用到的英文资源
  • zh_CN：模块内会用到的中午资源
  • rawfile ：各类资源文件。这类文件会被直接打包进应用，不经过编译

其他类配置文件

用于编译构建，包括构建配置文件、编译构建任务脚本、混淆规则文件、依赖的共享包信息等。

- build-profile.json5：工程级或Module级的构建配置文件，包括应用签名、产品配置等。

- hvigorfile.ts：应用级或Module级的编译构建任务脚本，开发者可以自定义编译构建工具版本、控制构建行为的配置参数。

- obfuscation-rules.txt：混淆规则文件。混淆开启后，在使用Release模式进行编译时，会对代码进行编译、混淆及压缩处理，保护代码资产。

UIAbility

UIAbility组件是一种包含UI的应用组件，主要用于和用户交互。

UIAbility 组件是系统调度的基本单元，类似于进程的概念。为应用提供绘制界面的窗口。一个应用可以包含一个或者多个UIAbility组件，例如支付应用中，可以将入口和收付款分别配置为独立的UIAbility。

具体表现形式：

- 每一个UIAbility组件实例都会在最近的任务列表中显示一个对应的任务

因此，对于开发者而言，可以根据场景选择单个或者多个UIAbility ，划分建议如下：

  • 对于开发者而言希望在任务视图中看到一个任务，那么就采用一个UIAbility+多个界面的形式，可以避免不必要的资源加载

  • 对于开发者而言希望在任务视图中看到多个任务，或者需要同时开启多个窗口，建议使用多个UIAbility实现不同功能

    例如在IM（即时通讯）应用中消息列表和音视频通话采用不同的UIAbility进行开发，那么实现应用的两个任务窗口在一个屏幕上分屏显示

任务视图：任务视图用于快速查看和管理当前设备上运行的所有任务或者应用

UIAbility的声明配置

为了应用能正常使用UIAbility，需要在module.json5配置文件中abilities标签中声明UIAbility的名称、入口、标签等相关信息。

生命周期

UIAbility的生命周期包括Create、Foreground、Background以及Destroy四个状态

Create状态

创建UIAbility实例，系统会调用onCreate() 回调函数，可以在回调函数中进行页面初始化操作。例如变量定义资源加载等，用于后续的UI展示。

Foreground状态

在UIAbility实例切换成前台时触发，系统会调用OnForeground()回调函数。

onForeground() 回调函数在UIAbility实例的UI可见之前，如UIAbility切换至前台时触发，可以在回调中申请系统需要的资源，或者重新申请在onBackground() 中释放的资源

Background状态

在UIAbility 实例切换成后台时调用，系统会调用onBackground() 回调函数。

onBackground() 回调函数在UIAbility 实例的UI完全不可见之后，如UIAbility 切换至后台时出发，可以释放在UI不可见时无用的资源，或者在此回调中执行较为耗时的操作，例如状态保存等。

拿定位应用举例，假设应用已经获取到定位授权权限。切换至前台，在UI显示之前在onForeground()回调中开启定位功能，获取到当前位置。切换至后台状态，在onBackground() 回调中关闭定位功能，以节省系统资源消耗

Destroy状态

在UIAbility 实例销毁时触发。系统会调用onDestroy() 回调函数。在回调函数中进行系统资源的释放、数据的保存等。

WindowStage事件（Stage）

当UIAbility实例创建之后，在进入Foreground 状态之前，系统会创建一个WindowStage。

WindowStage事件有以下几种状态，分别对应不同的事件情况

名称 值 说明
SHOWN 1 切到前台，无论是首次启动还是后台启动均会触发此状态
ACTION 2 获焦状态，应用窗口点击事件后的状态、应用启动后的状态
INACTION 3 失焦状态，打开新应用或者点击其他应用窗口时，原获焦窗口状态
HIDDEN 4 切到后台，应用上滑退出、应用窗口关闭
RESUMED 5 前台可交互状态，应用打开之后可以与用户交互的状态
PAUSED 6 前台不可交互状态，屏幕底部上划，应用进入到多任务界面的状态

该图来源于HarmonyOS官网

WindowStage有多个回调函数

  • onWindowStageCreate() ：当WindowStage创建完成后会进入该函数回调，可以在该函数回调中设置UI加载、设置WindowStage的事件订阅

  • onWindowStageWillDestroy()：在``WindowStage将要销毁时执行，释放WindowStage`对象获取的资源

  • onWindowStageDestroy：在UIAbility实例销毁之前，会先进入onWindowStageDestroy() 回调函数，可以在此回调函数中释放UI资源

启动模式

UIAbility的启动模式是指UIAbility实例在启动时不同呈现状态，系统提供了三个启动模式

- singleton（单例模式）

- multiton（多实例模式）

- specified（指定实例模式）

singleton模式

singleton模式是单例模式，也是默认的启动方式。

每次调用startAbility()方法时，如果应用进程中该类型的UIAbility实例已经存在，则复用系统中的该UIAbility实例。系统只存在唯一一个该UIAbility实例

场景模拟

如果应用的UIAbility实例已经创建，且设置为单例模式。再次调用startAbility()方法启动该UIAbility实例，启动的还是原来的UIAbility实例，并未重新新建一个UIAbility实例。所以此时只会进入onNewWant()方法，不会进入onCreate() 和onWindowStageCreate() 生命周期

使用方法

要使用singleton单例模式，只要在module.json5中launchType字段配置为singleton

multiton模式

multiton模式是多实例模式，每次调用startAbility()方法时，都会在应用进程中创建一个新的该类型UIAbility实例。也即在最近任务列表中可以看到有多个该类型的UIAbility实例

使用方法

要使用multiton多实例模式，只要在module.json5 中launchType字段配置为multiton

specified模式

specified启动模式也即指定实例模式，针对一些特殊场景使用（例如新建文档每次要希望打开新的文档实例，要打开一个已保存的文件时则希望打开同一个文件实例）

场景模拟

如果应用的UIAbility实例已经创建，且设置为指定实例模式。再次调用startAbility()方法启动该UIAbility实例，启动的还是原来的UIAbility实例，并未重新新建一个UIAbility实例。所以此时只会进入onNewWant()方法，不会进入onCreate() 和onWindowStageCreate() 生命周期

使用方法

1、要使specified指定实例模式，要在module.json5中lauchType字段配置为specified

2、在创建指定实例的UIAbility实例时，可以指定一个唯一的字符串Key，用于定位该实例。这样在调用startAbility()方法时，应用就可以根据指定的key来识别请求不同的UIAbility实例。

基本用法

UIAbility组件的基本用法包括几个方面：指定UIAbility的启动页面、获取UIAbility上下文UIAbilityContext以及终结UIAbility实例

指定UIAbility启动页面

应用的UIAbility在启动过程中，需要指定启动页面，否则系统会因为没有默认加载页面导致白屏。

可以在UIAbility的onWindowStageCreate()生命周期回调函数中，通过WindowStage对象的loadContent()方法设置启动页面。

获取UIAbility上下文

UIAbility类拥有自身的上下文信息，该信息为UIAbilityContext类的实例。UIAbilityContext类拥有abilityinfo、currentHapModuleInfo等属性。

如果需要在页面中获取当前Ability的Context，可以调用getContext()接口获取与当前页面所关联的UIAbilityContext或ExtensionContext

  • 在UIAbility中可以通过this.context获取UIAbility实例的上下文信息

  • 在页面中获取UIAbility实例的上下文信息

    • 需要导入依赖资源context模块
    • 组件中定义一个context变量

    也可以导入资源context模块之后，在具体使用UIAbilityContext前进行变量定义

终止UIAbility实例

需要终止UIAbility实例可以调用terminateSelf()方法实现

数据同步

基于当前模型，可以通过以下几种方式来实现UIAbility与UI之间的数据同步

- 使用EventHub

- 使用AppStorage/LocalStorage

EventHub

在基类Context中提供了EventHub对象，可以通过发布订阅方式实现事件传递。事件传递前，订阅者需要进行事件订阅，发布者发布事件订阅者收到事件并处理

EventHub为UIAbility组件提供事件机制，使它们能够进行订阅、取消订阅以及触发事件等通信功能。

AppStorage/LocalStorage

  • AppStorage是一个全局的状态管理器，适用于多个UIAbility共享同一状态数据的情况。
  • LocalStorage是一个局部的状态管理器，适用于单个UIAbility内部使用的状态数据。

组件内交互

UIAbility是系统调度的最小单元。在设备的功能模块之间跳转时，会涉及到启动特定的UIAbility。

启动应用内的UIAbility组件主要有以下几种方式：

- 启动应用内的UIAbility，不获取返回结果

- 启动应用内的UIAbility，获取返回结果

- 启动UIAbility的指定页面

启动应用内的UIAbility，不获取返回结果

1、在EntryAbility中，通过调用startAbility()方法启动UIAbility

  • wantUIAbility实例启动的入口参数
    • bundleName为待启动应用的Bundle名称
    • abilityName为待启动的Ability名称
    • moduleName在待启动的UIAbility属于不同的Module时添加
    • parameters为自定义信息参数

2、在FuncAbility的onCreate()或者onNewWant()生命周期回调文件中接收EntryAbility传递过来的参数。

在被拉起的FuncAbility中，可以通过获取传递过来的want参数的parameters来获取拉起方UIAbility的PID、Bundle Name等信息。

3、在FuncAbility业务完成之后，如需要停止当前UIAbility实例，在FuncAbility中通过调用terminateSelf()方法实现。

4、如需要关闭应用所有的UIAbility实例，可以调用ApplicationContext的killAllProcesses()方法实现关闭应用所有的进程。

启动应用内的UIAbility，获取返回结果

1、在EntryAbility中，调用startAbilityForResult()接口启动FuncAbility，异步回调中的data用于接收FuncAbility停止自身后返回给EntryAbility的信息。

2、在FuncAbility停止自身时，需要调用terminateSelfWithResult()方法，入参abilityResult为FuncAbility需要返回给EntryAbility的信息。

3、FuncAbility停止自身后，EntryAbility通过startAbilityForResult()方法回调接收被FuncAbility返回的信息，RESULT_CODE需要与前面的数值保持一致。

启动UIAbility的指定页面

一个UIAbility可以对应多个页面，在不同的场景下启动该UIAbility时需要展示不同的页面，例如从一个UIAbility的页面中跳转到另外一个UIAbility时，希望启动目标UIAbility的指定页面。

UIAbility的启动分为两种情况：UIAbility冷启动和UIAbility热启动。

- UIAbility冷启动：指的是UIAbility实例处于完全关闭状态下被启动，这需要完整地加载和初始化UIAbility实例的代码、资源等。

- UIAbility热启动：指的是UIAbility实例已经启动并在前台运行过，由于某些原因切换到后台，再次启动该UIAbility实例，这种情况下可以快速恢复UIAbility实例的状态。

调用方UIAbility指定启动页面

调用方启动另外一个UIAbility时，通常需要跳转到指定的页面。此时需要在传入的want参数中配置指定的页面路径信息，可以通过want中的parameters参数增加一个自定义参数传递页面跳转信息。

目标UIAbility冷启动

目标UIAbility冷启动时，在目标UIAbility的onCreate()生命周期回调中，接收调用方传过来的参数。然后在目标UIAbility的onWindowStageCreate()生命周期回调中，解析调用方传递过来的want参数，获取到需要加载的页面信息url，传入windowStage.loadContent()方法。

目标UIAbility热启动

在应用开发中，会遇到目标UIAbility实例之前已经启动过的场景，这时再次启动目标UIAbility时，不会重新走初始化逻辑，只会直接触发onNewWant()生命周期方法。为了实现跳转到指定页面，需要在onNewWant()中解析参数进行处理。

例如短信应用和联系人应用配合使用的场景。

  1. 用户先打开短信应用，短信应用的UIAbility实例启动，显示短信应用的主页。
  2. 用户将设备回到桌面界面，短信应用进入后台运行状态。
  3. 用户打开联系人应用，找到联系人张三。
  4. 用户点击联系人张三的短信按钮，会重新启动短信应用的UIAbility实例。
  5. 由于短信应用的UIAbility实例已经启动过了，此时会触发该UIAbility的onNewWant()回调，而不会再走onCreate()和onWindowStageCreate()等初始化逻辑。

具体操作步骤如下

1、冷启动短信应用的UIAbility实例时，在onWindowStageCreate()生命周期回调中，通过调用getUIContext()接口获取UI上下文实例UIContext对象。

2、在短信应用UIAbility的onNewWant()回调中解析调用方传递过来的want参数，通过调用UIContext中的getRouter()方法获取Router对象，并进行指定页面的跳转。此时再次启动该短信应用的UIAbility实例时，即可跳转到该短信应用的UIAbility实例的指定页面。
